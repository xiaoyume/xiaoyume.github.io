1 InnoDB页介绍

读取记录时，InnoDB存储引擎不是吧记录一条条从磁盘中读出（因为这样非常慢），将数据划分为若干个页，以页作为磁盘与内存之间交互的基本单位，InnoDB中页的大小为16KB。

行格式

我们平时存储数据的时候都是以记录的形式插入表中的。在数据库存储的格式称之为行格式。InnoDB中有4中行格式。分别为Compact,Redundant,Dynamic,Compressed。

Compact行格式

可以想象得到，一行记录必须有记录的数据和另外的信息保证可以解析这个数据。

变长字段列表是和后面记录的真实数据一一对应的，因为数据中会有varchar这种长度不定的数据，所以会有这个字段。数据是顺序放的，变长字段列表是逆序放置的。注意，非变长字段不在里面（char(10)）。

null值列表是表示字段是否为空的，会空出不为空的字段。顺序也是逆序和变长字段长度列表一致。

记录头信息是固定5个字节。

22 UNDO日志

事务回滚的需求

事务id

START TRANSACTION READ ONLY

START TRANSACTION READ WRITE

如果对表进行了 增、删、改操作，那么InnoDB存储引擎就会给它分配一个独一无二的事务id。

如果事务里只有查询操作是不会分配事务id的，事务id是唯一的

事务id是如何生成的

  • 服务器里维护一个全局变量，每当需要分配事务id时，酒后把这个变量作为事务id分配给该事务，并且把变量自增1。
  • 每当这个变量的值是256的倍数时，就会把变量的值刷新到系统表空间的页号为5的页面称之为Max Trx ID的属性处，这个属性占用8个字节的空间。（为什么设计256，就是记录这个值的考虑，因为系统启动时才有这个变量，如何不存起来或者值设置的过大，会丢失这个记录）
  • 系统启动时这个Max Trx ID值会加上256才赋值给全局变量。

trx_id列

行记录中都会记录这个trx_id信息

UNDO页面链表

单个事务链表

一个事务有多条语句，一个语句对若干条记录进行改动，对每条记录改动前，都需要记录1条或2条（更新主键，删除插入）undo日志，产生的undo日志可能一个页面放不下，需要放到多个页面

一个事务可能会有多种语句，产生不同类型的undo日志。所以会有两个undo页面的链表，一个称为insert undo链表，另一个称为update undo链表。

另外，InnoDB规定普通表和临时表改动产生的undo日志要分开记录，所以一个事务最多有4个undo页面链表。

当然不是一开始就会有4个undo链表，是按需分配。

多个事务Undo页面链表

不同事务执行过程中产生的undo日志需要被写到不同的undo页面链表中。

undo 日志的写入过程

segment段

段是逻辑上的一个概念，本质上是由若干零散页和若干完整的区组成。

undo log segment header

规定每个undo页面链表都对应一个段，称为undo log segment,链表中的页面都是在这个段里面申请的。

所以在undo页面链表的第一页，first undo page设计了一个Undo log segment header的部分，包含了链表对应的段segment header信息以及其它的一些关于这个段的信息。

  • TRX_UNDO_STATE 本Undo链表处于什么状态

    TRX_UNDO_ACTIVE

    TRX_UNDO_CACHED

    TRX_UNDO_TO_FREE

    TRX_UNDO_TO_PURGE

    TRX_UNDO_PREPARED

  • TRX_UNDO_LAST_LOG 最后一个undo log header的位置

  • TRX_UNDO_FSEG_HEADER 对应段Segment header信息

undo log header

每个undo页面链表是一个组，每写入一组undo日志时，会记录关于这个组的属性。

在第一个页面真正写入undo日志之前，会填充undo page header,undo log segment header, undo log header

总结

重用undo页面

大部分事务开启时只修改了几条记录，占用非常少的空间。

判断是否可重用

  • 链表只包含一个undo页面

  • undo页面占用空间小于整个页面3/4

  • insert undo链表

    事务提交后就没有用了，直接覆盖

  • update undo 链表

    不能直接覆盖，因为update undo还要用于mvcc

回滚段

为了管理undo链表，设计了Rollback segement header页面，存放了各个undo页面链表的first undo page。

从回滚段申请undo页面链表

初始，undo slot设置为FIL_NULL,表示不指向任何页面

一个rollback segment header页面包含1024个undo slot，可以对应1024个undo页面链表，如果1024个undo slot值都占用了，就会回滚这个事务，并且报错

多个回滚段

事务分配undo页面链表的详细过程

  • 事务在执行过程中对普通表的记录首次做改动之前，首先会到系统表空间的第5号页面中分配一个回滚段（其实就是获取一个Rollback Segment Header页面的地址）。一旦某个回滚段被分配给了这个事务，那么之后该事务中再对普通表的记录做改动时，就不会重复分配了。

      使用传说中的round-robin（循环使用）方式来分配回滚段。比如当前事务分配了第0号回滚段，那么下一个事务就要分配第33号回滚段，下下个事务就要分配第34号回滚段，简单一点的说就是这些回滚段被轮着分配给不同的事务（就是这么简单粗暴，没什么好说的）。

  • 在分配到回滚段后，首先看一下这个回滚段的两个cached链表有没有已经缓存了的undo slot，比如如果事务做的是INSERT操作，就去回滚段对应的insert undo cached链表中看看有没有缓存的undo slot；如果事务做的是DELETE操作，就去回滚段对应的update undo cached链表中看看有没有缓存的undo slot。如果有缓存的undo slot，那么就把这个缓存的undo slot分配给该事务。

  • 如果没有缓存的undo slot可供分配，那么就要到Rollback Segment Header页面中找一个可用的undo slot分配给当前事务。

      从Rollback Segment Header页面中分配可用的undo slot的方式我们上面也说过了，就是从第0个undo slot开始，如果该undo slot的值为FIL_NULL，意味着这个undo slot是空闲的，就把这个undo slot分配给当前事务，否则查看第1个undo slot是否满足条件，依次类推，直到最后一个undo slot。如果这1024个undo slot都没有值为FIL_NULL的情况，就直接报错喽（一般不会出现这种情况）～

  • 找到可用的undo slot后，如果该undo slot是从cached链表中获取的，那么它对应的Undo Log Segment已经分配了，否则的话需要重新分配一个Undo Log Segment，然后从该Undo Log Segment中申请一个页面作为Undo页面链表的first undo page。

  • 然后事务就可以把undo日志写入到上面申请的Undo页面链表了！

  对临时表的记录做改动的步骤和上述的一样，就不赘述了。不错需要再次强调一次，如果一个事务在执行过程中既对普通表的记录做了改动，又对临时表的记录做了改动，那么需要为这个记录分配2个回滚段。并发执行的不同事务其实也可以被分配相同的回滚段，只要分配不同的undo slot就可以了。
