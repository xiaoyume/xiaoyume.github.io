1 InnoDB页介绍

读取记录时，InnoDB存储引擎不是吧记录一条条从磁盘中读出（因为这样非常慢），将数据划分为若干个页，以页作为磁盘与内存之间交互的基本单位，InnoDB中页的大小为16KB。

行格式

我们平时存储数据的时候都是以记录的形式插入表中的。在数据库存储的格式称之为行格式。InnoDB中有4中行格式。分别为Compact,Redundant,Dynamic,Compressed。

Compact行格式

可以想象得到，一行记录必须有记录的数据和另外的信息保证可以解析这个数据。

变长字段列表是和后面记录的真实数据一一对应的，因为数据中会有varchar这种长度不定的数据，所以会有这个字段。数据是顺序放的，变长字段列表是逆序放置的。注意，非变长字段不在里面（char(10)）。

null值列表是表示字段是否为空的，会空出不为空的字段。顺序也是逆序和变长字段长度列表一致。

记录头信息是固定5个字节。

4 UNDO日志

事务回滚的需求

事务id

START TRANSACTION READ ONLY

START TRANSACTION READ WRITE

如果对表进行了 增、删、改操作，那么InnoDB存储引擎就会给它分配一个独一无二的事务id。

如果事务里只有查询操作是不会分配事务id的，事务id是唯一的

事务id是如何生成的

  • 服务器里维护一个全局变量，每当需要分配事务id时，酒后把这个变量作为事务id分配给该事务，并且把变量自增1。
  • 每当这个变量的值是256的倍数时，就会把变量的值刷新到系统表空间的页号为5的页面称之为Max Trx ID的属性处，这个属性占用8个字节的空间。（为什么设计256，就是记录这个值的考虑，因为系统启动时才有这个变量，如何不存起来或者值设置的过大，会丢失这个记录）
  • 系统启动时这个Max Trx ID值会加上256才赋值给全局变量。

trx_id列

行记录中都会记录这个trx_id信息

UNDO页面链表

单个事务链表

一个事务有多条语句，一个语句对若干条记录进行改动，对每条记录改动前，都需要记录1条或2条（更新主键，删除插入）undo日志，产生的undo日志可能一个页面放不下，需要放到多个页面

一个事务可能会有多种语句，产生不同类型的undo日志。所以会有两个undo页面的链表，一个称为insert undo链表，另一个称为update undo链表。

另外，InnoDB规定普通表和临时表改动产生的undo日志要分开记录，所以一个事务最多有4个undo页面链表。

当然不是一开始就会有4个undo链表，是按需分配。

多个事务Undo页面链表

不同事务执行过程中产生的undo日志需要被写到不同的undo页面链表中。

undo 日志的写入过程

segment段

段是逻辑上的一个概念，本质上是由若干零散页和若干完整的区组成。

undo log segment header

规定每个undo页面链表都对应一个段，称为undo log segment,链表中的页面都是在这个段里面申请的。

所以在undo页面链表的第一页，first undo page设计了一个Undo log segment header的部分，包含了链表对应的段segment header信息以及其它的一些关于这个段的信息。

总结

重用undo页面

大部分事务开启时只修改了几条记录，占用非常少的空间。

判断是否可重用

  • 链表只包含一个undo页面

  • undo页面占用空间小于整个页面3/4

  • insert undo链表

    事务提交后就没有用了，直接覆盖

  • update undo 链表

    不能直接覆盖，因为update undo还要用于mvcc

回滚段

为了管理undo链表，设计了Rollback segement header页面，存放了各个undo页面链表的first undo page。

从回滚段申请undo页面链表

初始，undo slot设置为FIL_NULL,表示不指向任何页面

一个rollback segment header页面包含1024个undo slot，可以对应1024个undo页面链表，如果1024个undo slot值都占用了，就会回滚这个事务，并且报错

多个回滚段

事务分配undo页面链表的详细过程

  • 事务在执行过程中对普通表的记录首次做改动之前，首先会到系统表空间的第5号页面中分配一个回滚段（其实就是获取一个Rollback Segment Header页面的地址）。一旦某个回滚段被分配给了这个事务，那么之后该事务中再对普通表的记录做改动时，就不会重复分配了。

      使用传说中的round-robin（循环使用）方式来分配回滚段。比如当前事务分配了第0号回滚段，那么下一个事务就要分配第33号回滚段，下下个事务就要分配第34号回滚段，简单一点的说就是这些回滚段被轮着分配给不同的事务（就是这么简单粗暴，没什么好说的）。

  • 在分配到回滚段后，首先看一下这个回滚段的两个cached链表有没有已经缓存了的undo slot，比如如果事务做的是INSERT操作，就去回滚段对应的insert undo cached链表中看看有没有缓存的undo slot；如果事务做的是DELETE操作，就去回滚段对应的update undo cached链表中看看有没有缓存的undo slot。如果有缓存的undo slot，那么就把这个缓存的undo slot分配给该事务。

  • 如果没有缓存的undo slot可供分配，那么就要到Rollback Segment Header页面中找一个可用的undo slot分配给当前事务。

      从Rollback Segment Header页面中分配可用的undo slot的方式我们上面也说过了，就是从第0个undo slot开始，如果该undo slot的值为FIL_NULL，意味着这个undo slot是空闲的，就把这个undo slot分配给当前事务，否则查看第1个undo slot是否满足条件，依次类推，直到最后一个undo slot。如果这1024个undo slot都没有值为FIL_NULL的情况，就直接报错喽（一般不会出现这种情况）～

  • 找到可用的undo slot后，如果该undo slot是从cached链表中获取的，那么它对应的Undo Log Segment已经分配了，否则的话需要重新分配一个Undo Log Segment，然后从该Undo Log Segment中申请一个页面作为Undo页面链表的first undo page。

  • 然后事务就可以把undo日志写入到上面申请的Undo页面链表了！

  对临时表的记录做改动的步骤和上述的一样，就不赘述了。不错需要再次强调一次，如果一个事务在执行过程中既对普通表的记录做了改动，又对临时表的记录做了改动，那么需要为这个记录分配2个回滚段。并发执行的不同事务其实也可以被分配相同的回滚段，只要分配不同的undo slot就可以了。

事务并发执行可能的问题

  • 脏写（Dirty Write）

一个事务修改了另一个事务修改未提交的数据，导致事务修改失效

  • 脏读

一个事务读到了另一个未提交事务修改过的数据

  • 不可重复读

事务A读取了两次数据，两次读取之间事务B修改了数据，导致两次读取的数据不一致

  • 幻读

事务A读取了两次数据，期间事务B插入了一些数据，导致后一次查询出现了新的记录

四种隔离级别

按照严重性来排序

脏写 > 脏读 > 不可重复读 > 幻读

隔离级别：

  • READ UNCOMMITTED 读未提交

  • READ COMMITTED 读已提交

  • REPEATABLE READ 可重复读

  • SERIALIZABLE 可串行化

    隔离级别 脏读 不可重复读 幻读
    READ UNCOMMITTED 可 可 可
    READ COMMITTED 不 可 可
    REPEATABLE READ 不 不 可
    SERIALIZABLE 不 不 不

脏写问题太严重了，丢失修改在那种隔离级别下都是不被允许的。

Mysql在可重复读隔离级别下，是可以禁止幻读问题的发生的。

MySQL默认的隔离级别是可重复读

6 MVCC原理

版本链

对于InnoDB存储引擎来说，它的聚簇索引记录中包含了两个必要的隐藏列

  • trx_id 事务id，事务对记录进行改动就会修改
  • roll_pointer: 修改时，会把旧的版本写入到undo日志中，这个列指向undo日志，可以通过这个找到记录修改前的信息。

//如果插入一条记录
INSERT INTO hero VALUES(1, '刘备', '蜀');

实际上insert undo只在事务回滚的时候起作用，事务提交后就没用了，占用的Undo Log Segment也会被系统回收。

每次对记录改动，都记录undo日志，每条日志也有roll_pointer属性（insert没有，因为没有更早的记录），可以把这些undo日志连起来，串成一个链表。

上面的版本链信息的结构只是示意图，具体undo日志的结构看undo日志一节，里面有更详细的介绍。

ReadView

对于读未提交隔离级别的事务来说。可以读到未提交事务修改过的记录，直接读取记录最新版本就可以。

对于使用SERIALIZABLE隔离级别的事务来说，使用了加锁来访问记录。

对于READ COMMITED和REPEATABLE READ隔离级别的事务来说，必须保证度高已经提交的事务修改的记录，如果一个事务修改了记录但是未提交，是不能读取最新版本的记录的，核心问题是如何判断版本链中的哪些版本是当前事务可见的。

提出了ReadView的概念

  • m_ids 表示生成ReadView是当前系统中活跃的读写事务的事务id列表

  • min_trx_id 表示生成ReadView时系统中活跃的读写事务中最小的事务id

  • max_trx_id 表示生成ReadView时系统应该分配给下一个事务的id

  • creator_trx_id 表示生成该ReadView的事务的事务id

有了上面这几个概念，在访问某条记录时，按照下面的步骤判断记录的某个版本是否可见。

  • 如果被访问版本的trx_id属性与ReadView中的creator_trx_id值相同，意味就是当前事务访问自己修改的记录，可以被访问

  • 如果trx_id属性小于ReadView中的min_trx_id值，表明生成该版本的当前事务在生成ReadView前已提交，所以可以被当前事务访问。

  • 如果trx_id大于max_trx_id, 说明生成该版本的事务在当前事务生成ReadView后才开启， 不可被当前事务访问。

  • 如果被访问的版本trx_id值在ReadView的min_trx_id和max_trx_id之间，需要判断一下trx_id属性是不是在m_ids列表中，如果在说明创建ReadView时生成该版本的事务还是活跃的，版本不可被访问。不在说明，已经提交，可以被访问。

如果某个版本对当前事务不可见，就顺着版本链找到下一个版本记录。

MVCC

所谓MVCC多版本并发控制，指的就是READ COMMITTED,REPEATABLE READ这两种隔离级别的事务在执行普通select操作时访问记录的版本链的过程，使得不同事务不同事物，读写操作并发执行，从而提高系统性能。

关于purge

  • insert undo在事务提交之后就可以被释放，而update还需要支持MVCC，不能立即删除。
  • 为了支持MVCC,对于delete mark操作来说，仅仅是在记录上打上一个删除标记，并未真正删除，delete和update是同一类undo日志。

7 锁

并发事务访问相同记录的情况

  • 读-读 并发事务读取相同记录

无影响

  • 写-写 并发事务对相同的记录做出改动

这种情况可能发生脏写的问题，任何一种隔离级别都不允许发生。如果多个未提交事务相继对一条记录做出改动，需要排队执行，这个排队过程就是通过锁来实现的。锁其实是内存中的结构，在事务执行前是没有锁的，也就是说一开始就是没有锁结构与记录进行关联的。

事务想要对一条记录做出改动的时候，首先会查看内存中有没有与这条记录关联的锁结构，当没有的时候就会在内存中生成一个锁结构与之关联。

锁结构里的关键信息：

  • trx信息：代表这个锁结构是在哪个事务生成的。

  • is_waiting: 表示事务是否在等待。

这个过程称为加锁成功。

事务T1提交之前，如果另一个事务也想对这条记录做出改动，那么先查看有没有锁结构与这条记录关联，发现已有，然后也生成一个锁结构与这条记录关联，不过锁结构的is_waiting属性设置为true,表示事务需要等待。这个过程就是加锁失败。

在事务T1提交之后，就会把该事务生成的锁结构释放掉，然后看还有没有别的事务在等待获取锁，发现了事务T2还在等待获取锁，把T2事务生成的锁结构的is_waiting属性设置为false。然后把该事务对应的线程唤醒，继续执行。

  • 写-读或读-写 一个事务读取，一个事务修改

这种情况可能发生脏读，不可重复读，幻读。

SQL标准规定不同隔离级别下发生的问题不一样

READ UNCOMMITTED 脏读，不可重复读，幻读都可能发生

READ COMMITTED 不可重复读，幻读可能发生，脏读不可发生

REPEATABLE READ 幻读可能发生，脏读和不可重复读不可发生

SERIALIZABLE 都不可发生

MySQL在REPEATABLE READ隔离级别实际上就可以解决幻读问题。

如何解决这些问题呢？

方案一：利用多版本并发控制（MVCC）,写操作进行加锁

方案二：读写操作都加锁

一致性读（Consistent Reads）

利用MVCC进行的读取操作称之为一致性读，或者快照读。

普通SELECT语句，在READ COMMITTED,REPETABLE READ隔离级别下都是一致性读。

锁定读（Locking Reads）

共享锁和独占锁

  • 共享锁， S锁。在事务读取一条记录前，需要获取记录的S锁
  • 独占锁，也称排他锁，Exclusive locks简称X锁

兼容性：只有S锁和S锁兼容

InnoDB存储引擎中的锁

表级锁

对表执行select、insert、delete、update时，不会为这个表添加标记的S锁或者X锁。

  • 表级IS锁，IX锁

  • 表级的AUTO-INC锁

表中的列可以添加AUTO_INCREMENT属性。自动赋值

自动递增赋值原理：

  • 采用AUTO-INC锁，执行插入语句时就在表级别加上一个AUTO-INC锁，然后为每条待插入记录的列分配递增之，执行完成后把AUTO-INC锁释放掉。一个事务持有AUTO-INC锁的过程中，其它事务的插入语句都要被阻塞来保证分配的递增值连续合理。
  • 采用一个轻量级的锁，插入时获取这个轻量级锁，然后生成本次插入语句需要用到的AUTO_INCREMENET列的值后，把该锁释放掉，兵不需要等到整个插入语句执行完。

常见行锁类型

  • Record Locks LOCK_REC_NOT_GAP, 正经记录锁

比如把number为8的记录加一个正经记录锁

  • Gap Locks

解决幻读问题，可以用MVCC解决，也可以采用加锁翻案解决。使用加锁翻案时有个问题，就是事务第一次执行读取操作的时候，那些记录并不存在，无所加记录锁。gap锁，

number值为8的记录加了gap锁，意味着不允许别的事务在number为8的记录前面的间隙插入新的记录，也就是（3，8）之间不允许插入。

这个gap锁的提出仅仅时为了防止插入幻影记录而提出的，对一条记录加gap锁，不影响其它事务加锁。

问题来了，如何在20记录之后的间隙该怎么办？如何阻止在（20， +∞）区间插入记录呢，

数据页中有两条记录：

  • Infimum记录，表示页面中的最小记录
  • Supremum记录，表示页面中的最大记录

所以要限制（20， +∞）插入记录，就在Supremum记录上加上一个gap锁

  • Next-Key Locks

记录锁和间隙锁的合体

  • Insert Intention Locks

事务在插入一条记录前需要判断该位置是不是被别的事务加了间隙锁，如果有插入操作需要等待，直到拥有gap锁的事务提交。设计规定事务在等待时也需要在内存中生成一个锁结构，表明事务想在某个间隙插入新记录，但是现在在等待。插入意向锁 LOCK_INSERT_INTENTION

如果事务T1在number为8的记录加一个gap锁，然后T2和T3分别向hero表中插入4，5两条记录，

锁结构中有一个type属性，表示锁的类型。

T1持有间隙锁，T2,T3处于等待状态。T1释放锁，T2,T3就能获得对应的插入意向锁（is_waiting改为false）,T2,T3之间不会相互阻塞，会同时获取到插入意向锁，插入意向锁并不会阻止别的事务继续获取该记录上的任何类型的锁。

  • 隐式锁

一般情况的insert操作时不加锁的。如果一个事务插入了一条记录，然会另一个事务对这条记录加锁，就可能发生脏读、脏写的问题。

事务id起作用了：

情景一：对于聚簇索引，有一个隐藏的trx_id列。当前事务插入了一条记录，trx_id值就表示当前事务的事务id，如果其它事务对该记录添加S锁或X锁时，首先会看记录的trx_id是否是当前的活跃id，如果是就帮当前事务创建一个X锁，然后自己进入等待状态。

情景二：对于二级索引，没有trx_id列，但是在二级索引页面的page Header部分有一个PAGE_MAX_TRX_ID属性，属性表示对该页面进行改动的最大事务id，如果小于当前最小活跃事务id，说明页面修改事务已提交，否则就要在页面中定位到对应的二级索引，回表找到对应的聚簇索引记录，重复情景一的做法。

由于事务id的存在，导致相当于加了一个隐士锁。别的事务对记录加锁时，会帮助当前事务生成一个锁结构，然后自己再生成一个锁结构进入等待状态。

InnoDB内存锁结构

在对不同记录加锁时，符合下面的条件：

  • 在同一个事务里进行加锁操作
  • 被加锁的记录在同一个页面
  • 加锁的类型一样
  • 等待状态一样

就可以放到一个锁结构中。

type_mode 32位 拆分为lock_mode,lock_type,rec_lock_type

  • 锁的模式lock_mode,占用低4位，

LOCK_IS 0

LOCK_IX 1

LOCK_S 2

LOCK_X 3

LOCK_AUTO_INC 4

  • 锁类型（lock_type）占用5-8位

LOCK_TABLE 16 第5位为1

LOCK_REC 32 第6位为1

  • 行锁的具体类型（rec_lock_type），只有lock_type为LOCK_REC时才能被分为更多的类型。

LOCK_ORDINARY 0表示next-key锁

LOCK_GAP 512

LOCK_REC_NOT_GAP 1024

LOCK_INSERT_INTENTION 2048

is_waiting属性也在这个type_mode里。

LOCK_WAIT 256(第9个bit位),表示is_waiting为true
